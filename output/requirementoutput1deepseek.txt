Based on the provided code files, here are the **functional and non-functional requirements** for the Cargo Tracking and Booking System (DDDSample): --- ## **1. Functional Requirements** ### **1.1 Cargo Booking & Management** 1. **Book New Cargo** - The system shall allow a user to book a new cargo by specifying: - Origin location (UN/LOCODE) - Destination location (UN/LOCODE) - Arrival deadline (date/time) - The system shall generate a unique tracking ID for each new cargo. - The cargo shall be stored in the system with a status of "not routed." 2. **Request Possible Routes** - The system shall provide a list of possible itineraries (routes) for a given cargo (by tracking ID). - Each itinerary shall consist of one or more legs, each leg including: - Voyage number - Load location and time - Unload location and time 3. **Assign Cargo to Route** - The system shall allow a user to assign a cargo to a selected itinerary. - Upon assignment, the cargo’s delivery status shall be updated accordingly. 4. **Change Cargo Destination** - The system shall allow a user to change the destination of an existing cargo. - The route specification shall be updated, and the cargo’s delivery status shall be recalculated. 5. **List All Cargos** - The system shall provide a list of all cargos with their current routing and status information. ### **1.2 Cargo Tracking & Inspection** 1. **Inspect Cargo** - The system shall inspect a cargo’s delivery progress based on its handling history. - The system shall update the cargo’s delivery status (e.g., misdirected, arrived). - The system shall trigger events if the cargo is misdirected or has arrived at its final destination. 2. **Track Cargo** - The system shall provide the current status of a cargo, including: - Tracking ID - Origin and destination - Current location - Estimated time of arrival (ETA) - Next expected activity - Handling history (events) - The system shall indicate if the cargo is misdirected. ### **1.3 Handling Event Management** 1. **Register Handling Event** - The system shall allow registration of handling events for a cargo, including: - Event type (LOAD, UNLOAD, RECEIVE, CLAIM, CUSTOMS) - Completion time - Location (UN/LOCODE) - Voyage number (for LOAD/UNLOAD events) - The system shall validate that the cargo, voyage, and location exist. - The system shall publish an event indicating the cargo was handled. 2. **Handle File Uploads** - The system shall periodically scan a designated directory for files containing handling event data. - The system shall parse each file line-by-line and register handling events. - Lines that fail to parse shall be moved to a separate directory for manual review. 3. **REST API for Handling Reports** - The system shall provide a REST endpoint (`/handlingReport`) to submit handling reports in JSON format. - The endpoint shall accept a list of tracking IDs, allowing batch registration of handling events. ### **1.4 Routing Service** 1. **Fetch Routes** - The system shall provide a routing service that returns possible itineraries for a given route specification. - The routing service shall integrate with an external pathfinding service (GraphTraversalService). ### **1.5 User Interfaces** 1. **Web Interface for Cargo Administration** - The system shall provide a web interface for: - Booking new cargos - Viewing and updating cargo details - Selecting and assigning itineraries - Changing cargo destinations - Listing all cargos 2. **Web Interface for Cargo Tracking** - The system shall provide a public-facing tracking interface where users can enter a tracking ID to view: - Current cargo status - Handling event history - Next expected activity 3. **REST API for Cargo Tracking** - The system shall provide a REST endpoint (`/api/track/{trackingId}`) that returns cargo tracking information in JSON format. ### **1.6 Event Notifications** 1. **Publish Application Events** - The system shall publish events for: - Cargo handled - Cargo misdirected - Cargo arrived at destination - Handling event registration attempt received - Events shall be delivered via JMS (or other messaging infrastructure) for asynchronous processing. ### **1.7 Data Management** 1. **Sample Data Generation** - The system shall include a sample data generator to pre-populate the database with sample cargos, voyages, locations, and handling events for demonstration and testing. --- ## **2. Non-Functional Requirements** ### **2.1 Performance & Scalability** - The system shall handle multiple concurrent cargo bookings and tracking requests. - The routing service shall respond with possible itineraries within a reasonable time frame (e.g., < 5 seconds). - The file upload scanner shall process files without blocking other operations. ### **2.2 Reliability & Availability** - The system shall ensure data consistency for cargo and handling event aggregates. - The system shall handle external service failures (e.g., routing service) gracefully and provide fallback behavior where possible. ### **2.3 Security** - The administrative web interface shall be protected (authentication/authorization not shown in code but implied for real-world use). - The public tracking interface and REST API shall be read-only and not expose sensitive information. ### **2.4 Maintainability & Extensibility** - The system shall follow Domain-Driven Design (DDD) principles, with clear separation of domain, application, and infrastructure layers. - The codebase shall be modular, allowing replacement of external services (e.g., routing, messaging) with minimal impact. ### **2.5 Usability** - The web interfaces shall be intuitive, providing clear feedback for user actions. - The REST APIs shall follow standard conventions and provide meaningful error messages. ### **2.6 Integration** - The system shall integrate with external systems via: - JMS for asynchronous event handling - REST APIs for handling report submission and cargo tracking - External routing service (GraphTraversalService) for itinerary calculation ### **2.7 Data Integrity** - The system shall enforce data validation (e.g., UN/LOCODE format, tracking ID uniqueness) at the domain level. - Database transactions shall ensure consistency for operations that update multiple aggregates. ### **2.8 Monitoring & Logging** - The system shall log key operations (e.g., cargo booked, handling event registered) for auditing and troubleshooting. - Errors during file parsing, event processing, or external service calls shall be logged with sufficient detail. --- ## **3. Domain-Specific Constraints & Rules** 1. **Cargo Lifecycle** - A cargo must have an origin and destination (different locations). - A cargo may be re-routed multiple times. - A cargo’s delivery status is recalculated after each handling event or routing change. 2. **Handling Events** - LOAD and UNLOAD events require a voyage. - RECEIVE, CLAIM, and CUSTOMS events prohibit a voyage. - Handling events must reference existing cargos, voyages, and locations. 3. **Itinerary Validation** - An itinerary is valid only if it satisfies the cargo’s route specification (origin, destination, and arrival deadline). 4. **Misdirection Detection** - A cargo is considered misdirected if it is in a location not on its itinerary. 5. **UN/LOCODE Format** - All location codes must follow the UN/LOCODE standard (two letters followed by three letters/numbers). --- ## **4. Assumptions & Dependencies** - The system assumes the existence of an external pathfinding service (`GraphTraversalService`) for route calculation. - The system uses a relational database (JPA/Hibernate) for persistence. - The system uses JMS (ActiveMQ) for asynchronous messaging. - The system is built with Spring Boot and Java. - Sample data is provided for demonstration purposes. --- **Summary**: This system is a **Cargo Tracking and Booking System** that allows users to book, route, and track cargo shipments.
