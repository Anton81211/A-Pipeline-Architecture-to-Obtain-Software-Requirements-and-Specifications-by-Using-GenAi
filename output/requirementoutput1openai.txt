Below is a structured set of functional and non‑functional requirements derived from the given code base. They describe what the system must do, not how it is implemented. --- ## 1. Cargo Booking & Routing ### 1.1 Book New Cargo - The system shall allow a user (booking clerk/UI) to register a new cargo into the tracking system. - Input: - Origin UN/LOCODE (string) - Destination UN/LOCODE (string) - Arrival deadline (date/time, UTC) - Behaviour: - Validate that origin and destination exist in the `LocationRepository`. - Validate that origin and destination are not the same. - Create a new unique `TrackingId`. - Create a `Cargo` with: - Origin = origin location - RouteSpecification = (origin, destination, arrival deadline) - No itinerary initially - Delivery snapshot derived with: - routing status = NOT_ROUTED - transport status = NOT_RECEIVED - no last event - no ETA - not unloaded at destination - not misdirected - Persist the new cargo in `CargoRepository`. - Return the tracking id of the new cargo. - Log that a new cargo has been booked. ### 1.2 List All Cargos - The system shall allow listing all registered cargos. - Each listed cargo shall be represented as a `CargoRoutingDTO` containing: - Tracking id - Origin UN/LOCODE - Final destination UN/LOCODE - Arrival deadline - Misrouted flag (true if routingStatus == MISROUTED) - Zero or more itinerary legs (each with voyage number, from/to UN/LOCODE, load/unload time). - Cargos are obtained from `CargoRepository.getAll()`. ### 1.3 View Cargo for Routing (Admin) - The system shall allow loading a single cargo by tracking id for routing purposes. - Output: `CargoRoutingDTO` as above. - If the cargo does not exist, the controller must treat this as an error (e.g. handled in UI; repository returns null). ### 1.4 Change Destination - The system shall allow changing the destination of an existing cargo. - Input: - Tracking id - New destination UN/LOCODE - Behaviour: - Fetch cargo by tracking id from `CargoRepository`. - Fetch new destination location by UN/LOCODE from `LocationRepository`. - Create a new `RouteSpecification` with: - origin = current cargo origin (unchanged) - destination = new destination - arrival deadline = existing arrival deadline of cargo’s route specification - Assign the new route specification to the cargo via `cargo.specifyNewRoute(...)`. - Recalculate the delivery snapshot synchronously (routing status, ETA, misdirection, transport status, etc.) based on the new route specification and current itinerary (if any). - Persist the updated cargo. - Log that the destination was changed. ### 1.5 Request Possible Routes - The system shall allow requesting possible itineraries (routes) for a cargo. - Input: - Tracking id - Behaviour: - Fetch the cargo by tracking id. - If no such cargo exists, return an empty list. - Call `RoutingService.fetchRoutesForSpecification` with the cargo’s current `RouteSpecification`. - Only itineraries that satisfy the `RouteSpecification` (origin, destination and arrival before deadline) shall be included in the result. - Output: - A list of `Itinerary` domain objects. - In the admin UI, these are converted to `RouteCandidateDTO`s (list of `LegDTO`s). ### 1.6 Assign Cargo to Route - The system shall allow assigning a cargo to a selected itinerary. - Input: - Tracking id - Itinerary (represented as `RouteCandidateDTO` in the UI, converted to domain `Itinerary`) - Behaviour: - Fetch the cargo by tracking id. - If the cargo does not exist, an error shall be thrown (IllegalArgumentException). - Assign the itinerary to the cargo via `cargo.assignToRoute(...)`, which: - Replaces the current itinerary legs. - Recalculates the delivery snapshot synchronously (routing status = ROUTED or MISROUTED, ETA, next expected activity, etc.). - Persist the updated cargo. - Log that the cargo was assigned to a new route. --- ## 2. Cargo Inspection & Delivery Status ### 2.1 Inspect Cargo - The system shall support a cargo inspection operation to update the delivery progress and fire notifications. - Input: - Tracking id - Behaviour: - Reject null tracking id. - Fetch cargo by tracking id. - If cargo is not found: - Log a warning. - Return without error. - Fetch `HandlingHistory` for this cargo from `HandlingEventRepository`. - Recalculate the cargo’s `Delivery` snapshot using `cargo.deriveDeliveryProgress(handlingHistory)`, which: - Determines last handling event (if any). - Determines transport status: - NOT_RECEIVED (no events) - IN_PORT (RECEIVE, UNLOAD, CUSTOMS) - ONBOARD_CARRIER (LOAD) - CLAIMED (CLAIM) - Determines last known location (location of last event, or UNKNOWN). - Determines current voyage (voyage of last event if ONBOARD_CARRIER, else NONE). - Determines routing status: - NOT_ROUTED (no itinerary) - ROUTED (itinerary exists and satisfies route specification) - MISROUTED (itinerary exists but does not satisfy route specification) - Determines misdirection: - false if no last event or no itinerary - else true if last event is not expected according to the itinerary rules. - Determines ETA: - itinerary’s final arrival date if cargo is on track (ROUTED and not misdirected) - else unknown. - Determines next expected activity (or null) based on: - No last event: expect RECEIVE at origin - After RECEIVE: expect LOAD of first leg at leg’s load location & voyage - After LOAD: expect UNLOAD at matching leg’s unload location & voyage - After UNLOAD: expect LOAD on next leg or CLAIM at final unload location - After CLAIM: no next activity. - Determines `isUnloadedAtDestination`: - true if last event is UNLOAD at the route specification’s destination. - If `delivery.isMisdirected()` is true: - Fire `ApplicationEvents.cargoWasMisdirected(cargo)`. - If `delivery.isUnloadedAtDestination()` is true: - Fire `ApplicationEvents.cargoHasArrived(cargo)`. - Persist the updated cargo. --- ## 3. Handling Events Registration ### 3.1 Register Handling Event (Synchronous Service) - The system shall support registering handling events for cargos. - Input: - completionTime (Instant) - trackingId - voyageNumber (may be null depending on event type) - unLocode - type (RECEIVE, LOAD, UNLOAD, CLAIM, CUSTOMS) - Behaviour: - Determine `registrationTime` as current time. - Use `HandlingEventFactory.createHandlingEvent(...)` to: - Look up cargo by trackingId; if not found, throw `UnknownCargoException`. - Look up voyage by voyageNumber if provided; if not found, throw `UnknownVoyageException`. - Look up location by unLocode; if not found, throw `UnknownLocationException`. - Validate that: - For LOAD or UNLOAD: a voyage must be present; otherwise an exception is thrown. - For RECEIVE, CLAIM, CUSTOMS: voyage must be absent; otherwise an exception is thrown. - Construct a `HandlingEvent` with the appropriate constructor (with/without voyage). - Store the new handling event via `HandlingEventRepository.store(event)`. - Publish `ApplicationEvents.cargoWasHandled(event)` so that: - A message is placed on the “cargo handled” JMS queue. - Downstream consumers can perform inspection/updates. - Log that the handling event was registered. - Transaction behaviour: - If a `CannotCreateHandlingEventException` (or its subclasses) is thrown, the transaction shall be rolled back. ### 3.2 Handling Event History - The system shall provide the handling history for a cargo: - For a given tracking id, `HandlingEventRepository.lookupHandlingHistoryOfCargo` shall: - Return a `HandlingHistory` containing all handling events for that cargo. - Provide: - `distinctEventsByCompletionTime()` = events without duplicates, ordered by completionTime ascending. - `mostRecentlyCompletedEvent()` = last event in that ordered list or null. - History queries must ignore events with null location (per JPA query constraint in JPA implementation). --- ## 4. Routing Service Integration ### 4.1 External Routing Service - The system shall integrate with an external graph traversal service to compute route candidates. - For a given `RouteSpecification`: - Extract origin and destination UN/LOCODEs and arrival deadline. - Call `GraphTraversalService.findShortestPath(originCode, destCode, limitations)` where: - `limitations["DEADLINE"]` = arrivalDeadline as ISO‑8601 string. - From each returned `TransitPath`, build an `Itinerary`: - Each `TransitEdge` yields a `Leg`: - Voyage = lookup by `VoyageNumber(edge.getEdge())` via `VoyageRepository`. - Load location = lookup by `UnLocode(edge.getFromNode())` via `LocationRepository`. - Unload location = lookup by `UnLocode(edge.getToNode())`. - Load/unload times = `edge.getFromDate()` / `edge.getToDate()`. - Filter out any itineraries that do not satisfy the original `RouteSpecification` (log a warning for those). - Return the list of valid itineraries. --- ## 5. Messaging and Asynchronous Processing ### 5.1 Application Events Abstraction The system shall provide an `ApplicationEvents` interface with methods: - `cargoWasHandled(HandlingEvent event)` - `cargoWasMisdirected(Cargo cargo)` - `cargoHasArrived(Cargo cargo)` - `receivedHandlingEventRegistrationAttempt(HandlingEventRegistrationAttempt attempt)` ### 5.2 JMS Implementation of Application Events - The JMS-based implementation shall: - On `cargoWasHandled`: - Log. - Send a text message with the cargo’s tracking id to the `cargoHandledQueue`. - On `cargoWasMisdirected`: - Log. - Send tracking id to `misdirectedCargoQueue`. - On `cargoHasArrived`: - Log. - Send tracking id to `deliveredCargoQueue`. - On `receivedHandlingEventRegistrationAttempt`: - Log. - Send an object message with the attempt to the `handlingEventRegistrationAttemptQueue`. ### 5.3 JMS Consumers The system shall provide JMS consumers that: - Listen on `CargoHandledQueue`: - Convert text message to tracking id. - Call `CargoInspectionService.inspectCargo(...)`. - Log parsing or processing errors. - Listen on `HandlingEventRegistrationAttemptQueue`: - Convert object message to `HandlingEventRegistrationAttempt`. - Call `HandlingEventService.registerHandlingEvent(...)` using data from the attempt. - Log errors. - Listen on `MisdirectedCargoQueue`, `DeliveredCargoQueue`, `RejectedRegistrationAttemptsQueue`: - Simply log received messages. ### 5.4 JMS Infrastructure - The system shall: - Configure an ActiveMQ connection factory using a `brokerUrl` property. - Configure named JMS destinations for: - `cargoHandledQueue` - `misdirectedCargoQueue` - `deliveredCargoQueue` - `handlingEventRegistrationAttemptQueue` - `rejectedRegistrationAttemptsQueue` - Configure a JMS template (`JmsOperations`) for sending messages. - Create and start a JMS `Connection` at startup. - Create a shared JMS `Session`. - Create message consumers for relevant queues and attach the appropriate `MessageListener`. --- ## 6. File‑Based Handling Reports ### 6.1 Directory Scanning - The system shall periodically scan an upload directory for text files containing handling event lines. - Configuration: - `uploadDirectory` and `parseFailureDirectory` must be different paths. - If they are the same, initialization shall fail. - Both directories shall be created if they do not exist; if creation fails, an exception shall be thrown. - Scheduling: - A scheduler shall run `UploadDirectoryScanner.run()` at a fixed rate (e.g. every 5 seconds), if the scanner bean exists. ### 6.2 Parsing Uploaded Files - For each file in the upload directory: - Read all lines. - For each line: - Split on two or more spaces: - If 5 columns: `[completionTime, trackingId, voyageNumber, unLocode, eventType]`. - If 4 columns: treat as `[completionTime, trackingId, "", unLocode, eventType]` (no voyage). - Otherwise, reject the line as invalid. - Parse: - completionTime: `"yyyy-MM-dd HH:mm"` in UTC. - trackingId: non-empty string. - voyageNumber: optional; if empty, treated as null. - unLocode: must be a valid UN/LOCODE pattern. - eventType: one of HandlingEvent.Type enum values. - If parsing succeeds: - Create a `HandlingEventRegistrationAttempt` with: - registrationTime = now - completionTime (parsed) - trackingId, voyageNumber, eventType, unLocode - Invoke `ApplicationEvents.receivedHandlingEventRegistrationAttempt(attempt)`. - If parsing fails: - Log the rejected line. - Collect the rejected line. - If there were rejected lines: - Append them to a file named `originalFilename + ".reject"` in the parseFailureDirectory. - If file processing completed (even with rejections): - Delete the original file. - If delete fails, log an error. - If an unexpected error occurs while processing a file: - Log the error. - Move the file to the parseFailureDirectory using `renameTo`; if that fails, log an error. --- ## 7. Web Service for Handling Reports (JSON API) ### 7.1 Submit Handling Report - Endpoint: - `POST /handlingReport` - Consumes and produces `application/json`. - Request body (`HandlingReport`): - `completionTime` (LocalDateTime, required) - `trackingIds` (list of tracking-id strings, required) - `type` (string, required; must map to HandlingEvent.Type) - `unLocode` (string, required; valid UN/LOCODE) - `voyageNumber` (string, optional) - Behaviour: - Validate request body (basic field presence via `@Valid`). - Parse: - completionTime -> Instant (UTC) - voyageNumber -> VoyageNumber or null - type -> HandlingEvent.Type - unLocode -> UnLocode - trackingIds -> list of TrackingId - For each trackingId: - Create a `HandlingEventRegistrationAttempt` with: - registrationTime = now - completionTime (parsed) - trackingId, voyageNumber, type, unLocode - Call `ApplicationEvents.receivedHandlingEventRegistrationAttempt(attempt)`. - If any unexpected exception occurs: - Log error. - Respond with HTTP 500 and error message. - On success: - Respond with HTTP 201 (Created) and no body. --- ## 8. Persistence and Domain Constraints ### 10.1 Cargo - A `Cargo` shall: - Have a unique `trackingId` string. - Have an origin (`Location`). - Have a `RouteSpecification` (origin, destination, arrivalDeadline). - Have zero or more `Leg`s as itinerary; if empty, `itinerary()` returns an empty `Itinerary`. - Have a `Delivery` snapshot that is updated: - On routing changes (route specification or itinerary changes) synchronously. - On handling events via `deriveDeliveryProgress(...)` using entire handling history. - Use `trackingId` as identity for equality and hash code. ### 10.2 Route Specification - A `RouteSpecification` shall: - Require non-null origin, destination, and arrivalDeadline. - Require origin and destination to be different (by identity). - `isSatisfiedBy(Itinerary)` is true only if: - Itinerary initialDepartureLocation == origin. - Itinerary finalArrivalLocation == destination. - Itinerary finalArrivalDate is strictly before arrivalDeadline. ### 10.3 Itinerary & Leg - An `Itinerary` shall: - Contain a non-empty, non-null list of `Leg`s when constructed. - Provide: - `initialDepartureLocation` (load location of first leg or UNKNOWN if no legs). - `finalArrivalLocation` (unload location of last leg or UNKNOWN). - `finalArrivalDate` (unload time of last leg or Instant.MAX if no legs). - Determine whether a `HandlingEvent` is expected based on: - RECEIVE: location must equal loadLocation of first leg. - LOAD: there must exist a leg with same loadLocation and voyage. - UNLOAD: there must exist a leg with same unloadLocation and voyage. - CLAIM: location must equal unloadLocation of last leg. - CUSTOMS: always allowed. - If there are no legs, all events are considered expected. - A `Leg` shall: - Have non-null voyage, loadLocation, unloadLocation, loadTime, unloadTime. - Be a value object (equality by all attributes). ### 10.4 HandlingEvent - A `HandlingEvent` shall: - Refer to a `Cargo`, a `Location`, completionTime, registrationTime, type, and optionally a `Voyage`. - Enforce type–voyage relationship: - LOAD/UNLOAD: voyage must be non-null; else IllegalArgumentException. - RECEIVE/CLAIM/CUSTOMS: voyage must be null; else IllegalArgumentException. - Be immutable after creation (except for JPA requirements). - Use cargo, voyage, completionTime, location, and type for equality. ### 10.5 Locations, Voyages, Schedules - `Location`: - Identified by unique UN/LOCODE. - Provides a static `UNKNOWN` instance for unknown locations. - `Voyage`: - Identified by unique voyage number. - Has a `Schedule` (one or more `CarrierMovement`s). - Provides static `NONE` voyage for unknown/no voyage. - `CarrierMovement`: - Has departure and arrival locations and times, all non-null. ### 10.6 Repositories - `CargoRepository`, `LocationRepository`, `VoyageRepository`, `HandlingEventRepository` shall: - Provide methods to find entities by their identity (tracking id, UN/LOCODE, voyage number). - Provide methods to store (persist) entities. - `CargoRepository` shall provide `nextTrackingId()` returning a unique id string (e.g. UUID-based). --- ## 11.
